DataGrid Component

A wrapper around MUI X DataGrid (Community) with app-wide defaults — sorting, filtering, column visibility, custom toolbar, expandable rows, and row grouping.

Note: This is a starting point only. Review, challenge, and improve based on real usage context. If something feels wrong — it might be. Use your judgment.
1

What We're Building & Why

Several screens need a data table with capabilities beyond basic sorting: column visibility toggle (show/hide columns per user preference), quick search, expandable rows with arbitrary content, and grouped rows (e.g. Rig → Well → Items). DataGrid is the shared component that covers all of these cases in a consistent, maintainable way.

It is a thin wrapper over MUI X DataGrid (Community, MIT licensed). The wrapper adds three app-specific concerns — styling defaults, localStorage persistence for column visibility, and a built-in toolbar slot — and passes everything else straight through to DataGrid.

DataGrid

Path: organisms/data-display/DataGrid/DataGrid.tsx

App-specific props (added on top of DataGridProps)

storageKey string — persists column visibility to localStorage under dg:{storageKey}:cols. Restored on mount. Use a unique key per page, e.g. "ncr-list". Omit if the screen doesn't need persistence.
searchable boolean — shows a search input in the toolbar. Uses DataGrid's built-in filterModel.quickFilterValues. Searches all visible columns.
toolbarActions ReactNode — right side of the toolbar. Used for primary CTAs (e.g. "+ New NCR", filter buttons). Toolbar is rendered only when this or searchable is set — no empty toolbar bar.

Key DataGrid props — commonly used in this project

rows* R[] — flat array of row objects. Each row must have a stable id field (or use getRowId).
columns* GridColDef[] — column definitions. Import type from @mui/x-data-grid.
loading boolean — shows a loading overlay. Pass directly from your query's isLoading state.
onRowsScrollEnd () => void — fires when the user scrolls to the last visible row. Use for infinite scroll (trigger fetchNextPage here).
initialState.columns
.columnVisibilityModel Record<string, boolean> — default column visibility on first load. Only effective when storageKey is not set (storageKey overrides this).
getDetailPanelContent ({ row }) => ReactNode — renders an expandable panel below a row. Content can be anything: a form, a nested table, a key-value block. Must also set getDetailPanelHeight. Requires DataGridPro if used — see note below.
rowGroupingModel string[] — groups rows by the given field(s), e.g. ['rig', 'well']. Shows collapsible group headers. Requires DataGridPremium if used — see note below.
ℹgetDetailPanelContent and rowGroupingModel are DataGridPro/Premium features. They are listed here because they are the desired patterns for screens like the open-order list (Rig → Well → Items). If those screens are confirmed in scope, the package must be upgraded. Do not implement a manual workaround — escalate the decision first.

Usage Examples

Case 1 — Simple flat table

Minimal — no toolbar, no custom column visibility, just render data.

import { DataGrid } from '@cx/ui';
import type { GridColDef } from '@mui/x-data-grid';

const columns: GridColDef[] = [
  { field: 'no',        headerName: 'No.',        width: 60 },
  { field: 'itemCode',  headerName: 'Item Code',  width: 130 },
  { field: 'qty',       headerName: 'Qty',        width: 80,  type: 'number' },
  { field: 'date',      headerName: 'Date',       width: 120, type: 'date' },
  { field: 'rig',       headerName: 'Rig',        flex: 1 },
  { field: 'well',      headerName: 'Well',       flex: 1 },
  { field: 'uom',       headerName: 'UOM',        width: 80 },
  { field: 'unitPrice', headerName: 'Unit Price', width: 110, type: 'number' },
];

<DataGrid columns={columns} rows={rows} loading={isLoading} />

Case 2 — Toolbar with search, column visibility, and action button

Most common pattern. Column selection is persisted to localStorage via storageKey.

const columns: GridColDef[] = [
  { field: 'ncrNo',     headerName: 'NCR No.',          width: 140 },
  { field: 'title',     headerName: 'Title',            flex: 1 },
  { field: 'status',    headerName: 'Status',           width: 120,
    renderCell: ({ value }) => <StatusChip status={value} /> },
  { field: 'assignee',  headerName: 'Lead Investigator', width: 160 },
  { field: 'category',  headerName: 'Category',          width: 130 },
  { field: 'rootCause', headerName: 'Root Cause',        width: 160 },
];

<DataGrid
  columns={columns}
  rows={rows}
  loading={isLoading}
  storageKey="ncr-list"
  searchable
  toolbarActions={<Button variant="contained" onClick={onNew}>+ New NCR</Button>}
  initialState={{
    columns: {
      // default — show only these 4; user can toggle the rest via Columns button
      columnVisibilityModel: { category: false, rootCause: false },
    },
  }}
/>
→Default to hiding secondary columns via columnVisibilityModel. What "secondary" means must come from actual users — not from showing everything and hoping people hide it. When storageKey is set, the user's own selection overrides this default after their first visit.

Case 3 — Truncated text column with tooltip

For columns with long text values (title, description, root cause). Fix maxWidth to prevent one long value from pushing other columns.

import { Tooltip } from '@mui/material';

const columns: GridColDef[] = [
  { field: 'ncrNo', headerName: 'NCR No.', width: 140 },
  {
    field: 'title',
    headerName: 'Title',
    maxWidth: 240,
    flex: 1,
    renderCell: ({ value }) => (
      <Tooltip title={value} placement="top-start">
        <span style={{ overflow: 'hidden', textOverflow: 'ellipsis', whiteSpace: 'nowrap' }}>
          {value}
        </span>
      </Tooltip>
    ),
  },
  { field: 'status', headerName: 'Status', width: 120 },
];

<DataGrid columns={columns} rows={rows} />
!Never truncate ID or reference columns (NCR No., Transfer No., Lot No.). These are scannable identifiers — users need to read them in full at a glance.

Case 4 — Infinite scroll (load more on scroll)

Use onRowsScrollEnd instead of a "Load More" button. Works with TanStack Query's useInfiniteQuery.

const { data, isLoading, isFetchingNextPage, hasNextPage, fetchNextPage }
  = useInfiniteNcrList(params);

const rows = data?.pages.flatMap(p => p.data) ?? [];

<DataGrid
  columns={columns}
  rows={rows}
  loading={isLoading || isFetchingNextPage}
  onRowsScrollEnd={() => { if (hasNextPage) fetchNextPage(); }}
  scrollEndThreshold={200}   // px before bottom to trigger — tune per screen
/>

Case 5 — Row action buttons

For tables that need Accept / Edit / More actions on each row. Use renderCell on a dedicated actions column.

const columns: GridColDef[] = [
  // ...other columns
  {
    field: '__actions',
    headerName: '',
    width: 120,
    sortable: false,
    align: 'right',
    headerAlign: 'right',
    renderCell: ({ row }) => (
      <Box sx={{ display: 'flex', gap: 0.5 }}>
        <Button size="small" variant="contained" onClick={() => onAccept(row.id)}>
          Accept
        </Button>
        <OverflowMenu items={rowMenuItems(row)} />
      </Box>
    ),
  },
];

<DataGrid
  columns={columns}
  rows={rows}
  disableRowSelectionOnClick   // prevent accidental selection when clicking buttons
/>

Implementation Notes

Default config baked into DataGrid

Applied unconditionally — every screen gets these without having to pass them.

Hardcoded defaults

density "compact" — matches design system table density.
disableRowSelectionOnClick true — rows are not "selected" by a click unless the screen explicitly needs checkboxSelection. Pass disableRowSelectionOnClick={false} to override per screen.
sx — column header Font size 11px, uppercase, letter-spacing 0.06em, grey.50 background. Matches the rest of the design system table headers.
sx — cell Font size 13px, row border divider color.

storageKey — column visibility persistence

On mount, read localStorage.getItem('dg:{storageKey}:cols') and merge into initialState.columns.columnVisibilityModel. On every onColumnVisibilityModelChange, write the updated model back. Wrap the localStorage read in a try/catch — corrupted JSON must not crash the page.

// Pattern inside DataGrid.tsx
const savedVisibility = storageKey
  ? (() => {
      try {
        return JSON.parse(localStorage.getItem(`dg:${storageKey}:cols`) ?? 'null');
      } catch { return null; }
    })()
  : null;

// Merge: saved overrides initialState default
initialState={{
  ...props.initialState,
  columns: {
    ...props.initialState?.columns,
    columnVisibilityModel:
      savedVisibility ?? props.initialState?.columns?.columnVisibilityModel,
  },
}}
!initialState is read only once on mount — it is intentionally uncontrolled. Do not use a controlled columnVisibilityModel prop here; that would fight the user's stored selection on every re-render.

sx prop — merging with defaults

MUI's sx prop accepts an array. Use this pattern so a caller can pass their own sx without overriding the component's defaults:

<DataGrid
  sx={[
    defaultSx,                               // component defaults
    ...(Array.isArray(sx) ? sx : [sx]),      // caller overrides
  ]}
/>

Hard Constraints

✗Do not re-export DataGrid types through @cx/ui. Callers import GridColDef, GridRowParams, etc. directly from @mui/x-data-grid. Only DataGridProps is exported from @cx/ui.
✗No hardcoded colors or magic numbers. MUI theme tokens only in sx blocks: grey.50, divider, text.secondary, etc.
✗No any, no console.log. Component must be properly generic: DataGrid<R extends GridValidRowModel>. Explicit return type on all exports.
✗No @cx/api or @cx/auth imports. This is a UI component — it receives data via props, never fetches.
!Don't rebuild what DataGrid already provides. No custom sort logic, no custom pagination, no custom column menu implementation. Pass standard DataGrid props. The wrapper's job is defaults + three app-specific props only.
!Max 150 lines per file. DataGrid.tsx should sit near 100 lines. Split early.
→Atomic design: organism layer. May import from @mui/x-data-grid, @mui/material, and atoms/molecules from @cx/ui. No upward imports.
→Every public component must have JSDoc + @example with explicit return type.
