App Layout System

Reusable shell components — AppShell, AppHeader, SidePanel, Logo, UserAvatarMenu, NotificationButton, PageContainer + Sidebar nav type structure

Note: This is a starting point only. Review, challenge, and improve based on real usage context. If something feels wrong — it might be. Use your judgment.
1

What We're Building & Why

Each app currently has its own layout code — sidebar duplicated in apps/ops, no reusable header, no right panel pattern. This ticket builds pure display components in @cx/ui so every app can compose the same building blocks.

The result must match the reference screenshot. Key observations from the actual UI:

  • Dark navy header spans full width — logo left, Ask AI button + bell icon + user avatar right. No center slot used in apps/cx.
  • Sidebar collapsed (~40px wide), icon-only, teal icons on dark navy background. Sits flush left.
  • Content area has a light gray background (not white) — confirm the exact token with design.
  • Breadcrumb appears at the very top of the content area (white strip), above the gray content zone.
  • No visible page title (h1) in the default state — title prop on PageContainer should be optional.
  • Right panel (SidePanel) is closed by default — not visible unless Ask AI is clicked.
→Only apps/cx MainLayout gets refactored here. apps/portal and apps/ops will be migrated in separate tickets.

Sidebar Nav Structure (Existing Component)

The Sidebar component already exists — it is not new. This ticket only needs to wire it into AppShell via the sidebar prop. However the menuItems data structure needs to support the 3-level navigation seen in Figma. Clarify with the team whether the type already supports this or needs extending.

→Two states from Figma: Collapsed = icon only (64px wide). Expanded = icon + text label + chevron for accordion items. Hover or click toggles expansion.

Nav item data structure — 3 levels

Based on Figma. Verify against actual type in codebase before implementing.

NavItem type (proposed)

idstring — unique key.
labelstring — text shown when expanded. Also used as tooltip when collapsed.
iconReactNode — shown in both collapsed and expanded states.
hrefstring — navigation target. Omit if item is a parent accordion (has children).
childrenNavGroup[] — if present, item becomes an accordion. Cannot have href at the same time.

NavGroup type (sub-menu, 2nd level)

groupLabelstring — section header label (e.g. "Inspection", "NCR", "HSE"). Rendered as non-clickable divider text.
itemsNavSubItem[] — list of leaf items under this group.

NavSubItem type (3rd level — leaf)

idstring
labelstring — e.g. "Request Record", "NCR", "Schedule"
hrefstring — navigation target.
// Example menuItems structure
const navItems: NavItem[] = [
  {
    id: 'quality',
    label: 'Quality & HSE',
    icon: <QualityIcon />,
    children: [
      {
        groupLabel: 'Inspection',
        items: [
          { id: 'req-record', label: 'Request Record', href: '/quality/inspection/request' },
          { id: 'quarantine', label: 'Quarantine Batch', href: '/quality/inspection/quarantine' },
        ],
      },
      {
        groupLabel: 'NCR',
        items: [
          { id: 'ncr', label: 'NCR', href: '/quality/ncr' },
        ],
      },
    ],
  },
  {
    id: 'dashboard',
    label: 'Dashboard',
    icon: <DashboardIcon />,
    href: '/',          // leaf item — no children
  },
];

Sidebar behavior summary

State Width What's visible Interaction
Collapsed (default) 64px Icon only. No label, no chevron. Hover/click icon → tooltip shows label. Click → expands sidebar.
Expanded 280px Icon + label. Accordion items show chevron (▼/▲). Click accordion item → reveals sub-menu with group labels + leaf items.
Sub-menu open 280px Group labels (non-clickable) + leaf items indented below. Click leaf → navigate. Active leaf gets teal left border + highlight.
!Check the existing Sidebar type before adding new fields. If NavItem already supports children in some form, extend it — don't create a parallel structure.

Hard Constraints

✗No @cx/auth or @cx/api imports in @cx/ui. Pass user info + handlers via props. Same pattern as the existing Sidebar.
✗Atomic hierarchy is strict. Atoms → no molecule/organism imports. Molecules → no other molecule/organism imports. Only organisms may import molecules.
✗No hardcoded colors or spacing. Theme tokens only. No hex values, no magic px numbers.
!Max 150 lines per file. Split early at ~120 lines — e.g. extract SidePanelHeader.tsx.
!No any, no console.log. Explicit return types on all exports. JSDoc + @example on all public components.
→AppHeader: position="sticky" not "fixed". Fixed removes the header from document flow — content renders underneath it.
→SidePanel persistent: must push main area. When open, AppShell must reduce the main flex width — not let SidePanel overlap content.

Build Order

1

Logo (atom)

No deps. Start here to learn the folder convention: Logo/Logo.tsx, Logo/index.ts, Logo/Logo.test.tsx.

2

NotificationButton + UserAvatarMenu

Only import atoms + MUI. Write component → write test → export from folder index.ts before moving on.

3

AppHeader + SidePanel + PageContainer

Can build AppHeader and SidePanel in parallel. Confirm Breadcrumb molecule exists in @cx/ui before starting PageContainer.

4

AppShell

Only after all other organisms are done. AppShell composes via props/slots — no direct sub-component imports.

5

Update barrel exports

Add all new components + types to packages/ui/src/index.ts. Run npm run biome:check immediately after.

6

Refactor apps/cx MainLayout

Replace hardcoded layout with composed shell. Visual smoke test against screenshot. Then open apps/portal + apps/ops and confirm no regressions.
