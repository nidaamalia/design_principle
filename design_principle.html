Tubestream Design Principles

Internal · For Designers & Dev · MUI v7
⚙️
MUI v7 context. All patterns and code references in this document use Material UI v7. Key things to know:
  • Use slots and slotProps for component customization — TransitionComponent, TransitionProps, and similar v5/v6 props have been removed.
  • MUI's spacing base is 8px (theme.spacing(1) = 8px). This document maps all spacing values to their theme.spacing() equivalent.
  • Typography variants map directly to MUI's built-in scale — see Section 12.
  • Density is controlled via component size prop: "small" | "medium" | "large".

Core Principles

Tubestream is a B2B supply chain management platform used by operational professionals — not casual users. The UI must prioritize efficiency, clarity, and confidence over aesthetics. Every design decision should be answerable with a single question:

💡
"Does this help users complete their work faster and with greater confidence?"
If not, remove it or simplify it.

🎯 Clarity First

Every action must communicate its outcome before it's clicked. Labels must be specific — not "Submit" but "Submit NCR". No ambiguity allowed.

⚡ Efficiency

Tubestream users are operators performing the same tasks repeatedly. Reduce cognitive load. Don't force users to read before they can act.

🛡 Safety Net

Destructive actions (delete, terminate, reject) must require confirmation. Reversible actions can execute immediately. Always distinguish between the two.

📐 Consistency

The same patterns must apply across all modules — Inspection, NCR, Transfer, Call Off. A user who learns one module should immediately understand the next.

Button Hierarchy

Contained · Outlined · Text · Icon

Three button levels: Contained (primary/CTA), Outlined (secondary), and Text (tertiary/inline). Each page should have a maximum of 1 Contained button in the primary CTA area. If two contained buttons appear side by side, one of them is a design mistake.

Contained
Outlined
Text
Icon Button

When to Use Which?

Contained — The page's primary action. The single most important CTA. Examples: "+ New Transfer", "Submit NCR", "Accept". Max 1 per section/form footer.
Outlined — Secondary action that matters but isn't the primary CTA. Examples: "Save as Draft", "Export XLSX", "Assign Role". Can appear alongside a Contained button.
Text Button — Tertiary action or link-like behavior. Used inside content, or for optional/non-critical actions. Examples: "Do It Later", "View Details", "Cancel".

Button Label Rules

Maximum 3 words for a button label. Ideal is 1–2 words. If more is needed, reconsider the design.
Verb + Noun, specific — "Submit NCR" not "Submit". "Export XLSX" not "Download".
Avoid ambiguous words: "OK", "Yes", "Confirm" without context. Replace with outcome-describing labels: "Accept NCR", "Delete Transfer".
Never use "Submit" alone in forms across Tubestream's many modules. Always clarify: "Submit Review", "Submit Enquiry", "Submit Transfer".

Button Sizes

Default (Page-level actions)

Use default size (height ~34px) for toolbars and main page headers.

Small (In-table / Inline)

Use Small for buttons inside table rows or dense panels.

Brand Color Usage in Buttons

Maintain brand consistency while preserving semantic clarity in action hierarchy. Brand colors should complement, not compete with, semantic colors for user actions.

What Are Semantic Colors? (Why They Matter)

🎨
Semantic colors = universal color meanings that users already understand. These colors have been standardized across the web for decades. When users see red, they instinctively think "danger" - not because of your brand, but because every app they've used follows this convention.

Real-World Examples Users Already Know

🔴 Red = Danger/Error (Universal)
❌ Gmail: Delete, Report Spam
❌ Slack: Remove, Leave Channel
❌ Jira: Delete Issue, Cancel
❌ Windows: Delete File, Empty Recycle
🟢 Green = Success/Go (Universal)
✅ Gmail: Send, Archive
✅ Slack: Send Message, Join
✅ Jira: Approve, Resolve
✅ Windows: Install, Complete
💡 User psychology: When users see red button, they pause and think "Is this safe?" - this built-in safety mechanism protects them from mistakes.

Overflow Menu (3+ Actions)

When a single row has 3 or more actions, hide them behind a kebab menu (⋯). Surface the most frequently used action(s) directly (max 1–2), and place the rest in the overflow menu.

⚠️
Don't render 4–5 small buttons inline in a table row. It creates visual noise, makes focus difficult, and is unusable on mobile.

Overflow Menu Rules

Destructive actions always at the bottom of the overflow menu, styled in red to visually differentiate them.
Max 5–6 items in an overflow menu. If more are needed, consider grouping with a divider.
Never put the primary CTA inside an overflow menu. If "Dispatch" is the most important action, it must be visible directly.

When to Use Icon-Only Buttons?

Icon buttons are acceptable in tables only if the icon is universally recognizable (edit ✏️, delete 🗑, download ⬇️) and a tooltip appears on hover. Never use custom or ambiguous icons without a label.

Icon + Tooltip (✓)

Universal icons + tooltip on hover. Saves space while remaining accessible.

Ambiguous Icons (✗)

Function is unclear. Users have to guess. Always pair with a tooltip or label.

Navigation & Links

Distinguish between navigating to another page (use a Link or Text Button), triggering an action within the page (use a Contained/Outlined Button), and external links (distinct color + icon).

Navigation Hierarchy

Primary navigation (between modules) → Sidebar menu.
Navigation to detail (from list to detail page) → Click the row OR click a text link on the ID/Name.
Navigation to a related sub-page → Text Button with arrow: View NCR Report →
Don't use a Contained button for pure navigation. Contained buttons = actions that change state (submit, approve, delete). Navigation = links or text buttons.

Row-Click Pattern

For lists/tables where the primary intent is "open the detail", make the entire row clickable. This is more efficient than requiring users to click a "View" button. Row-click must provide visual feedback (cursor: pointer + hover background change).

Tubestream has many lists that need to reveal detail without a full page navigation. Use expandable rows for lightweight detail, and detail pages for deep content.

Default for all Tubestream forms: validate on submit, then clear each field's error as the user fixes it. This avoids showing red errors on untouched fields which feels aggressive.

MUI DataGrid — Pagination Standard

For pages using MUI DataGrid, pagination is built-in. Standardize the configuration across all modules as follows:

// Standard DataGrid pagination config for all Tubestream list pages
<DataGrid
  rows={rows}
  columns={columns}
  // Default 25 rows — enough to scan without overwhelming
  initialState={{ pagination: { paginationModel: { pageSize: 25 } } }}
  // Consistent options across all modules
  pageSizeOptions={[25, 50, 100]}
  // Always server-side for Tubestream (data is too large for client-side)
  paginationMode="server"
  rowCount={totalRowCount}
  onPaginationModelChange={handlePaginationChange}
  // Always small density
  density="compact"
  // Disable row count estimate — always show exact count
  rowCountMode="exact"
/>
Default page size is 25 across all modules. Never let individual developers decide — inconsistent defaults break user expectations when navigating between modules.
Always use server-side pagination for Tubestream data. Client-side pagination (loading all rows then paginating in the browser) is only acceptable for lists with a known hard cap under ~200 rows.

Modals & Confirmations

Three Modal Types in Tubestream

📋 Form Modal

For lightweight input. Examples: "Assign Role & Task", "Add Widget", "Set Due Date". Footer: Cancel (text) + Submit (contained).

⚠️ Confirmation Modal

For destructive or irreversible actions. Examples: "Terminate Transfer", "Delete NCR". Must re-confirm with a specific action label.

📤 Export/Download Modal

Filter before exporting. Example: Export NCR List (select Period, Type, Status). Primary button: "Export XLSX".

Confirmation Modal Rules

Don't put the destructive button on the right by default! Position: left = "Cancel" (outlined/text), right = destructive action (contained red). Ensure Cancel is easy to find to avoid mis-clicks.
The modal title must describe the action, not just "Are you sure?" — Use: "Terminate Transfer TR-2025-042?" or "Delete NCR-001 permanently?"
No modal on top of a modal. If a nested action is needed from within a modal, consider an inline flow or a multi-step modal instead.

Confirmation Message Format & Structure

All confirmation modals follow a standardized message structure to ensure clarity and consistency across the system.

Message Template

Title:
[Action Verb] [Data Name/ID]?
Body:
Want to [Action] [Data Name]?
This action cannot be undone.
Examples:
• Title: "Delete NCR-001 permanently?"
Body: "Want to delete NCR-001? This action cannot be undone."
• Title: "Terminate Transfer TR-2025-042?"
Body: "Want to terminate Transfer TR-2025-042? This action cannot be undone."
• Title: "Reject Order PO-2025-156?"
Body: "Want to reject Order PO-2025-156? This action cannot be undone."
Title format: [Action Verb] [Data Name/ID]? — Be specific. "Delete NCR-001?" not "Are you sure?". Always include the data identifier so users know exactly what they're about to do.
Body format: "Want to [action] [data name]? This action cannot be undone." — Reinforce the action and data, then warn about irreversibility. Keep it concise.
For reversible actions, omit the warning. If the action can be undone (e.g., "Archive", "Pause"), remove "This action cannot be undone." and replace with context-specific guidance if needed.

⏳ Loading State

Purpose: Indicate system activity without blocking understanding. Layout must be preserved — no content jump when data arrives.

🦴 Skeleton (preferred)

Use for content-heavy pages: list pages, tables, detail pages. Preserves layout, sets user expectation of content shape.

🌀 Spinner (limited use)

Use only for action-triggered loading: submitting a form, exporting a file, a single inline operation. Not for full page or table load.

Exporting…
Table skeleton (✓)

Layout is preserved. User knows what's coming.

Full-page spinner (✗)

No layout, no context. User has no idea what's loading or how much content to expect.

Use <Skeleton> for tables and content areas. Match the skeleton shape to real content — same number of rows, same column widths. Don't show 3 skeleton rows if the real list usually has 20.
Loading transitions must be automatic — never require the user to click "Refresh" to see loaded content. When data arrives, content replaces skeleton without any user action.
Never block the entire page with a spinner or overlay unless the action genuinely requires it (e.g. file upload affecting the whole session). Partial loading (table loading while toolbar is usable) is always better.

📭 Empty State

Purpose: Explain absence of data and guide the user's next action. This is not an error — it's an expected condition for new modules or unused features.

📋
No NCRs yet
Non-conformance reports will appear here once they are created. Start by raising your first NCR.
Never show "No data" alone. Always include: what's missing, why it might be missing, and a clear next action.
Illustration/icon is optional — keep it low visual weight. A simple emoji or outlined icon is fine. Avoid heavy illustrations that distract on a professional tool.
The CTA in an empty state is the same as the page's primary CTA. Don't invent a new action — it's always "+ New [Record]".
Only show empty state when the dataset truly has zero records — not when a filter is applied. A filtered empty result is a "No Results" state (different pattern below).

🔍 No Results State

Purpose: Explain why nothing matches the current filter/search criteria, and give a fast exit. This is distinct from Empty State — data exists, but the current query returns nothing.

Status: Completed Date: Jan 2025
🔍
No NCRs match these filters
Try adjusting your filters or search terms.
Always show which filters are currently active in the no-results state so users understand why nothing matches. Show them as chips above the empty message.
"Clear Filters" is the primary CTA — always Outlined or Contained. Not a plain text link, because this is the most likely recovery action.
Never show the page's normal primary CTA (+ New NCR) in a No Results state. The issue is the filter, not the absence of data. Don't conflate the two.

❌ Error State

Purpose: Communicate failure and give a clear recovery path. Never show raw system errors or stack traces. Every error must be human-readable and actionable.

Blocking vs. Non-Blocking

🚧 Blocking Error

The entire page or section cannot load. Replace content area with error message.

Example: failed to load NCR list, session expired, server 500.

⚠️ Non-Blocking Error

A specific action failed but the page is still usable. Show as inline alert or toast.

Example: export failed, form submission failed, file upload rejected.

⚠️
Failed to load NCR list
There was a problem connecting to the server. Your data is safe.
Error code: 503 — Service Unavailable
❌
Export failed
The file could not be generated. Please try again or contact support if the issue persists.
Always include a recovery action. At minimum: "Try Again" for retriable errors, "Go Back" for dead-end pages, "Contact Support" for unrecoverable failures.
Never show raw error codes alone. Show the human-readable message first, error code secondary and small. Operators don't know what "ERR_NETWORK_CHANGED" means.
Error must not persist after recovery. If user retries successfully, clear the error state completely — don't leave a stale error banner above fresh content.
Define blocking vs. non-blocking per error type in the codebase. A failed export should never blank out the entire page. Scope error display to the smallest affected surface.

Unauthorized state must align with RBAC rules. If a user's role changes, the state should reflect the new permissions on next navigation — not require a full logout/login.
